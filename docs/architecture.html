Factory Inventory Management

Architecture Reference — full-stack demo with Vue 3 frontend, Python FastAPI backend, and in-memory mock data

System Architecture

Three-Layer Architecture

The application is split into three distinct layers: a browser-based Vue 3 single-page application running on port 3000, a Python FastAPI backend on port 8001, and a flat-file data layer consisting of JSON files loaded into memory at startup. There is no database — all persistence is in-process.

Data Flow

Filter Selection to API Response

All four global filters are held as Vue refs in a singleton composable. Views watch these refs and re-fetch whenever any value changes.

1

User changes a filter in FilterBar.vue

FilterBar.vue binds each <select> to a reactive ref via v-model. The four refs — selectedPeriod, selectedLocation, selectedCategory, selectedStatus — live in client/src/composables/useFilters.js as a module-level singleton, so all views share the same state automatically.

2

View detects the change and calls getCurrentFilters()

Each view (e.g. Orders.vue, Dashboard.vue) imports useFilters() and sets up a watch on the filter refs. On change it calls getCurrentFilters(), which maps selectedPeriod to the month API param and returns a plain object: { warehouse, category, status, month }.

3

api.js builds and fires the HTTP request

client/src/api.js receives the filters object. It uses URLSearchParams to append only non-'all' values, then issues an axios GET to http://localhost:8001/api/<endpoint>?<params>. Example: GET /api/orders?warehouse=London&status=shipped.

4

FastAPI route handler applies filters in-memory

server/main.py receives the query params as typed Optional[str] arguments. The route calls apply_filters(items, warehouse, category, status) — a plain list comprehension that skips items not matching the value. For time-based endpoints it then calls filter_by_month(items, month), which supports both direct month strings (2025-06) and quarter keys (Q2-2025) via QUARTER_MAP.

5

Pydantic validates and serialises the response

The route is decorated with response_model=List[Order] (or the appropriate model). FastAPI runs each dict through the Pydantic model, strips unexpected fields, and returns a JSON array. Pydantic models are defined in main.py: InventoryItem, Order, DemandForecast, BacklogItem, PurchaseOrder.

6

View stores raw data in refs, derived state in computed

The axios response resolves and the view stores the array in a reactive ref (e.g. allOrders, inventoryItems). Aggregations such as totals, counts, and chart data are expressed as Vue computed properties that derive from those raw refs, so the DOM updates automatically.

API Reference

API Endpoints

All endpoints are served by FastAPI on http://localhost:8001. Filter parameters are optional query strings; omitting them (or passing all) returns all records.

GET /api/inventory

Returns all inventory items. Supports warehouse and category filtering. Does not support month/time filtering (inventory has no time dimension).

warehouse (optional) category (optional)
GET /api/inventory/{item_id}

Returns a single inventory item by its id field. Returns HTTP 404 if not found.

No query parameters
GET /api/orders

Returns all orders. Supports all four filters. Month values accept both direct month strings (e.g. 2025-06) and quarter keys (e.g. Q2-2025).

warehouse (optional) category (optional) status (optional) month (optional)
GET /api/orders/{order_id}

Returns a single order by its id field. Returns HTTP 404 if not found.

No query parameters
GET /api/dashboard/summary

Returns aggregate KPI values: total inventory value, low-stock count, pending orders count, total backlog count, and total orders value. Applies all four filters to both inventory and orders before computing.

warehouse (optional) category (optional) status (optional) month (optional)
GET /api/demand

Returns all demand forecast records. No filtering supported.

No query parameters
GET /api/backlog

Returns all backlog items, each augmented with a has_purchase_order boolean derived by joining against the purchase orders list. No filtering supported.

No query parameters
GET /api/spending/summary

Returns aggregate spending summary statistics.

No query parameters
GET /api/spending/monthly

Returns month-by-month spending breakdown.

No query parameters
GET /api/spending/categories

Returns spending broken down by product category.

No query parameters
GET /api/spending/transactions

Returns recent transaction line items.

No query parameters
GET /api/reports/quarterly

Derives quarterly performance from orders: total orders, revenue, delivered count, avg order value, and fulfillment rate. Covers Q1–Q4 2025.

No query parameters
GET /api/reports/monthly-trends

Derives month-over-month order count, revenue, and delivered count from the orders data.

No query parameters
POST /api/purchase-orders

Creates a new purchase order. Accepts a CreatePurchaseOrderRequest body: backlog_item_id, supplier_name, quantity, unit_cost, expected_delivery_date, and optional notes.

No query parameters — JSON body required
GET /api/purchase-orders/{backlog_item_id}

Returns the purchase order associated with a given backlog item ID.

No query parameters
